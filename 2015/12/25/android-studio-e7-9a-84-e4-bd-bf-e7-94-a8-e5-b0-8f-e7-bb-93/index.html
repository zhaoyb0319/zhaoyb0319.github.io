Android studio的使用小结

　　前段时间一直忙于私事，再加上公司产品迭代，都没时间发文章了，不过现在基本稳定了，所以抽点时间做个小总结吧。Android Studio,基于IntelliJ IDEA,类似 Eclipse,,前段时间使用过IntelliJ，所以再使用AndroidStudio可以说很容易上手。
对于这个IDE呢，和大多数人的说法一样啦，就是不用不知道，一用离不了呀，所以后面重点说下，对于我来说，它给我带来的方便。

###主要是以下两点：

编码方面

  • 颜色、图片在布局和代码中可以实时预览
  • string可以实时预览
  • 可以直接打开文件所在位置
  • Android 专属的重构和快速修复
  • 功能强大的布局编辑器，可以让你拖拉 UI 控件并进行效果预览
  • 自动保存，无需一直Ctrl + S

打包方面

  • 基于[Gradle]的构建支持
  • Build Variants(创建构建支持）
  • Build Types(构建变种版本)
  • Product flavors(不同定制的产品）

用几个需求来说明它的方便之处

  1. 项目中使用到了友盟、友盟推送、第三方SDK等，需要设置各个相关的 key
  2. 测试环节，测试工程师需要各环境状态下的apk(比如，测试环境、仿真环境等）
  3. 开发需要不同类型的apk是否输入bug,是否进行混淆

　　当然每一类需求，单独处理时，也许可以通过代码呀，变量等记录，但当综合在一起时，就会出现打包时，改很多地方的变量，，，

　　用过ios的xcode编辑器，另外本身c中是含有预编译等指令，，所以只需要在xcode中设编译类型 模拟器会自动根据选择的类型生成不同配置的ipa以供测试

　　如今有了android stuio基本上等同于xcode了，，，
　　比如需要指定以下变量
　　BUILD_DEBUG(应用开启调试） = true|false
　　BUILD_API_URL(应用请求地址) = “http://192.168.0.1“ | “http://xxx.xxx.xxx.xxx“
　　UMENG_CHANNEL(友盟渠道）,
　　UMENG_MESSAGE_SECRET(友盟推送）
　　直接贴上我的 build.gradle

　　LOG_DEBGU为自定义的buildConfigField代码中可以获取配置的变量来决定是否开启调试manifestPlaceholders 是我一直所期待的，可以根据编译类型动态修改相关的key甚至是字段
　　另外代码也可通过buildType的值来决定使用哪个API_URL
　　设置都OK后，只需要手动选择studio的build variants

　　就可以直接进行真机调试了，，，假如你的项目中用到了友盟分享功能，，都知道 debug.keystore时无法使用微信分享等功能，，
　　必须出正式包（实际上是用正式签名），，因此就可以设置

　　signingConfig指向正式的签名，，这样就算是真机调试，直接刷入手机的版本也可以直接测试分享等功能

　　最后给出个人建议：
　　1. keystore持续使用同一个（保持和最终线上用一个），免得出现安装时签名不同导致的重新安装
　　2. 涉及到友盟的key和推送等第三方key时，，除非是release版本，，其他情况尽量用自己的，这样不会导致乱七八糟的数据统计到友盟上
　　3. 项目框架搭建之前就去设置好所有字段、变量，不要等到最后测试结束上线时，再调整这些，以免疏漏

总之一点，要做到：“万事俱备、只欠代码编译完，测试ok”
